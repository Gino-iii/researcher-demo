零、先理解你要复刻的是什么

原项目跑一次研究的完整链路（这是整个简版的骨架，建议先对照源码走读一遍）：

用户 query
  │
  ├─ ① choose_agent        用 LLM 给研究员定"角色人设"（可选功能）
  ├─ ② plan_research       先做一次初步搜索 → 让 LLM 基于搜索结果生成 3~5 个子查询
  │
  ├─ ③ 对每个子查询并发执行：
  │      搜索(retriever) → URL 去重(visited_urls) → 并发抓取(scraper)
  │      → 上下文压缩(embedding 相似度过滤，只留相关段落)
  │
  ├─ ④ 汇总所有子查询的 context
  └─ ⑤ write_report        把 context + 报告 prompt 喂给 SMART_LLM，流式生成 Markdown 报告

对应源码入口：GPTResearcher.conduct_research() 在 gpt_researcher/agent.py:371，真正的循环在 gpt_researcher/skills/researcher.py:95（"整个项目的心脏"），写报告在 agent.py:522 → skills/writer.py:49。

简版的取舍原则：只保留上面这条主链路。原项目里的 MCP、多检索器聚合、deep_research、multi_agents（LangGraph）、向量库、本地文档、图片生成，全部第一版不做，留作后期 DIY。

二、里程碑拆解

每一步都给出：做什么 → 参考源码 → 验收标准 → 坑。严格按顺序做，每步验收通过再进下一步。

M0：脚手架 + 配置层 半天

建目录、pyproject.toml、.env.example。写 config.py：一个 dataclass，从环境变量读配置，字段抄原项目默认配置里用得到的十来个：FAST_LLM、SMART_LLM、TEMPERATURE、MAX_SEARCH_RESULTS_PER_QUERY=5、MAX_ITERATIONS=3（子查询数）、TOTAL_WORDS=1200、BROWSE_CHUNK_MAX_LENGTH=8192、SIMILARITY_THRESHOLD=0.42、USER_AGENT、LANGUAGE。

  • 参考：gpt_researcher/config/variables/default.py:3、gpt_researcher/config/config.py
  • 注意：FAST/SMART 双模型的区分要保留——这是控制成本的核心，生成子查询/摘要用便宜的 FAST_LLM，写报告用 SMART_LLM。

验收：python -c "from mini_researcher.config import Config; print(Config())" 能打印从 .env 读到的配置。

M1：LLM 轻量层 半天

llm.py 写两个函数：chat(messages, model, temperature, stream, on_chunk) 和 chat_json(messages, model)（在 chat 之上加 JSON 解析 + 修复重试）。

  • 参考：gpt_researcher/utils/llm.py、gpt_researcher/llm_provider/generic/base.py
  • 坑 1：OpenAI 兼容接口兼容度参差——有些不支持 response_format=json_object，流式 chunk 的 delta.content 可能是 None。chat_json 别依赖 response_format，靠 prompt 约定 + 解析兜底。
  • 坑 2：LLM 返回 JSON 常带 ```json 围栏、尾逗号、单引号。原项目三层防御见 agent_creator.py:71（handle_json_error）和 :115（extract_json_with_regex）。至少要有"剥围栏 + json_repair"两层。
  • 顺手加成本追踪：响应里的 usage 字段是现成的，加一个模块级 token 累加器（十来行代码），从第一次调用起就看得见每轮研究烧多少 token（对应原项目 utils/costs.py，不必等到 M9）。

验收：chat_json 稳定拿到 dict；流式模式逐 token 打印。

M2：Tavily 检索器 半天

retrievers.py：class TavilyRetriever 带 async def search(query, max_results) -> list[{"href","body"}]。用 httpx.AsyncClient 直接调 Tavily REST——别用同步 SDK，否则在 event loop 里阻塞，M5 的 asyncio.gather 并发名存实亡；顺便少一个依赖，和 M3 技术栈统一。返回结构跟原项目对齐。

  • 参考：gpt_researcher/retrievers/tavily/tavily_search.py、接口约定 retrievers/utils.py
  • 顺手定义"检索器基类"（哪怕只有一个实现），这层抽象是原项目能塞进 18 个搜索引擎的原因。
  • 坑 3：Tavily 免费档 1000 次/月，一次研究 ≈ 5-6 次调用。加磁盘缓存（query 哈希 → JSON 文件），开发期省 90% 配额。

验收：search("量子计算 2025 进展") 返回 5 条带 URL 和摘要的结果。

M3：抓取器 1 天 · 坑最多

scraper.py：scrape_urls(urls) -> list[{"url","raw_content"}]。httpx 异步并发（信号量限 10）+ BeautifulSoup 提正文。

  • 参考：scraper/beautiful_soup/beautiful_soup.py、scraper/scraper.py、并发管理 skills/browser.py:36
  • 坑 4：必须处理超时（10s 硬超时）、非 HTML content-type（PDF/图片跳过）、编码错误（信 httpx 探测）、反爬 403（带 USER_AGENT，失败即丢弃，别死磕重试）。
  • 坑 5：正文可能几十万字符，按 BROWSE_CHUNK_MAX_LENGTH 截断，否则 M5 撑爆上下文。
  • 兜底：Tavily search 结果里本身带摘要（可选 include_raw_content 拿正文），被 403/超时干掉的 URL 可退回用它顶上，报告不至于因反爬缺料。

验收：给 5 个混合 URL（含 404、超慢站），15 秒内返回，失败的被静默过滤。

M5：研究循环 1-2 天 · 核心

async def conduct_research(query) -> list[str]:
    initial = await retriever.search(query)
    sub_queries = await plan_sub_queries(query, initial)   # 含原 query 本身
    contexts = await asyncio.gather(*[process(sq) for sq in sub_queries])
    return flatten(contexts)

async def process(sub_query):
    results = await retriever.search(sub_query)
    new_urls = self._dedupe(results)          # 全局 visited_urls 去重
    pages = await scrape_urls(new_urls)
    return await compress(sub_query, pages)
  • 参考（逐行对照读）：skills/researcher.py:95（conduct_research）、:311（_get_context_by_web_search，asyncio.gather 并发）、:519（_process_sub_query）、:851（_get_new_urls 去重）、:930（_scrape_data_by_urls）；子查询生成 query_processing.py:37 和 :112（把初搜结果塞进 prompt，值得保留）
  • 坑 8：visited_urls 集合被并发协程读写。单线程 event loop 下 set 安全，但"检查+添加"要在同一同步块完成，别 await 穿插。另外它要放在编排类实例上而非模块级全局——M8 一个进程跑多次研究时全局集合会串。
  • 坑 9：某子查询整体失败时返回空列表继续，别抛异常终止全局。
  • 用 on_event: Callable 回调贯穿全流程（比原项目一路透传 websocket 更干净），M8 换成 WebSocket 推送。

验收：日志能看到"生成 N 个子查询 → 每个搜 M 条 → 抓取成功 K 页"，最终拿到非空 context 列表。

M6：上下文压缩 1 天 · 可先丐版

推荐顺序：M5 完成后先用丐版直通 M7，端到端跑出第一份报告（尽早暴露超时/超限/编码等全链路问题），再回头做正式版压缩。

网页切 chunk → 算 embedding → 只保留与 sub_query 余弦相似度 > 0.42 的 chunk，取 top-k。

  • 丐版（先跑通）：每页截前 N 字符，总量控制在 SMART_LLM 上下文的 60%。
  • 正式版：llm.py 加 embed(texts)，context.py 切 chunk（1000 字符 + 100 重叠）、算余弦、过阈值排序取 top。numpy 甚至纯 Python 足够，不需要向量库。
  • 参考：context/compression.py、skills/context_manager.py:36、memory/embeddings.py
  • 坑 10（选型相关）：DeepSeek 没有 embedding 端点，Qwen/智谱/SiliconFlow 有。已确认：embedding 走阿里百炼（DashScope 兼容模式，text-embedding-v3），config 里 embedding 的 base_url/key 已与 chat 分开配置。
  • 坑 10b：embedding 请求有单批条数上限（DashScope text-embedding-v3 一批 ≤ 10 条），而一次研究可能切出几百个 chunk——embed(texts) 必须分批循环调用。

验收：压缩后 context 降到原来 10-30%，保留段落确实与子查询相关。

三、通用注意事项

  1. 全程 async。从 agent 到 scraper 全 async，否则并发抓取和流式输出都做不了。网络 IO 必须 async。
  2. 调试期控制成本：搜索缓存 + 子查询数临时调成 2 + FAST_LLM 用最便宜模型。一次全流程 ≈ 10-20 次 LLM 调用。
  3. 每层独立可测：M1-M6 每层写一个 tests/ 冒烟脚本，出问题能单独定位。
  4. 原项目 actions/ 是无状态纯函数、skills/ 是持有 researcher 引用的有状态类；简版可简化成模块函数 + 一个编排类。

四、时间预估

M0-M4 约 2-3 天，M5-M7 约 3-4 天（难点在 M5、M6），M8 约 1-2 天。总计 1.5-2 周业余时间到完整可用。
